# Доменное описание API

Контракт в TypeSpec: [main.tsp](main.tsp) (**REST**-ориентированные пути, подключена библиотека **@typespec/rest** для соглашений TypeSpec). Сгенерированный OpenAPI: [openapi/openapi.yaml](openapi/openapi.yaml) (после `npm run compile` или `make contracts`).

## Роли и продуктовые допущения

- **Владелец календаря** — единственный заранее заданный профиль; сценарии владельца в **административной** части приложения. Регистрации и входа в систему нет.
- **Гость** — любой пользователь публичной части без аккаунта; только бронирование слотов из разрешённого окна.

## Сущности (концептуальная модель)

### Владелец (CalendarOwner)

Не обязательно отдельный ресурс REST: это **режим приложения «админка»** и согласованный префикс **`/owner`**, под которым лежат ресурсы «коллекция типов событий» и «бронирования».

Семантика: управление типами событий и просмотр заявок. Идентичность «владельца» задаётся на уровне развёртывания/UI (не дублировать здесь техническое решение — по необходимости одна строка в AGENTS.md).

### Тип события (EventType)

Шаблон встречи, который задаётся в админке.

| Поле (логика)       | Описание                          |
|---------------------|-----------------------------------|
| `id`                | Стабильный идентификатор типа     |
| `title`             | Название                          |
| `description`       | Описание                          |
| `durationMinutes`   | Длительность в минутах для слота этого типа |

### Слот (AvailableSlot)

**Производная сущность:** интервал времени `[startAt, endAt)`, в который ещё **нет** конфликтующего бронирования ни у одного типа события, и который попадает в **окно гостевой записи** (ближайшие 14 дней от текущей даты на стороне сервера или согласованной политики кэша времени).

Слоты не описывают «ещё одну таблицу на бекенде» логически; их отдаёт субресурс **`.../available-slots`** при запросе по типу и диапазону дат при соблюдении правил календаря.

### Бронирование (Booking)

Факт записи гостя на конкретный интервал времени под выбранный тип события.

Логические поля: связь с типом события, интервал времени совпадающий с типовой длительностью для выбранного `eventTypeId`, опционально **контакт/метка для созвона** (без полноценного «профиля» пользователя). Точная форма контактных полей — в OpenAPI (см. модели `GuestBookingRequest` / `Booking`).

## Публичный сценарий гостя (поток)

1. Получить список типов событий с полями для отображения карточек брони (название, описание, длительность).
2. Выбрать тип, запросить **доступные слоты** на интервал календаря (клиент обычно ограничивает запрос диапазоном внутри 14-дневного окна; сервер **обязан** отвергнуть создание записи за пределами окна и при пересечении с занятым временем любого типа).
3. Создать бронирование на выбранный `startAt` и `eventTypeId`.

## Сценарии владельца (проверочный список к контракту)

| Сценарий | Покрытие в спецификации (см. [openapi/openapi.yaml](openapi/openapi.yaml)) |
|---------|--------------------------|
| Просмотр типов в админке | `GET /owner/event-types` |
| Создание типа события | `POST /owner/event-types` |
| Получить один тип | `GET /owner/event-types/{eventTypeId}` |
| Изменение / удаление типа | `PATCH`, `DELETE /owner/event-types/{eventTypeId}` |
| Единый список предстоящих встреч | `GET /owner/bookings` (query `from`, `to`) |
| Публичный список типов | `GET /guest/event-types` |
| Свободные слоты (субресурс) | `GET /guest/event-types/{eventTypeId}/available-slots` |
| Создание брони гостем | `POST /guest/bookings` |

## Инварианты (бизнес-правила для реализации)

1. На один и тот же момент/интервал **нельзя** две активные записи, даже если типы событий различны (глобальная занятость).
2. **Окно гостевой записи по умолчанию:** интервалы, на которые можно оформить бронирование через публичный API — только внутри **14 календарных дней**, начиная с **текущей даты** (интерпретация «сегодня» и таймзона — зона ответственности бэкенда; в контракте зафиксированы требования к запросам и кодам ошибок).

Коды ошибок в OpenAPI включают `409` для конфликта слота или пересечения с существующей записью и `422`/`400` для нарушения окна записи или неверных параметров — см. описания операций.
